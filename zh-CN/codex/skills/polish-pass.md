# 精修通道：设计结束的质量关卡 (Polish Pass: End-of-Design Quality Gate)

在将设计展示给利益相关者或发布之前，运行全面的质量检查。**一个经过精修的设计和一个未经精修的设计，是执行在不同用心程度上的同一个想法 —— 而这其中的差距，正是人们真正看到的。**

这项技能是四项更具体的审查技能的总和。在交付前将其作为最后一关来使用。

## 阶段 1：确认范围 (Phase 1: Confirm scope)

确定要精修的内容：

1. 用户刚刚完成或询问的 HTML 文件。
2. 项目中当前的主要交付物。
3. 如果不清楚，请询问。

阅读该文件。注意：
- 媒介（幻灯片 / 页面 / 移动端 / 仪表板 / 动画）
- 部署的上下文（内部审查 / 面向客户 / 营销）
- 任何用户申明的约束条件或范围边界

如果设计明显处于半成品状态（布局崩坏、部分缺失、用户仍在迭代中的占位内容），请说明情况并询问他们是想现在进行精修，还是等结构确定后再进行。

## 阶段 2：运行四次审查通道 (Phase 2: Run four review passes)

Codex 作为单个代理循环运行，因此请对文件**依次**运行这四次审查，并随时收集发现的问题。每个通道等同于其独立的审查技能，且作用域仅限此文件。不要在发现第一个问题后就停下来 —— 在进行修复之前完成所有四个通道，以便修复阶段能够纵观全局。

### 通道 1：无障碍性审计 (Pass 1: Accessibility audit)

运行完整的 `accessibility-audit` 审查：
- 对比度和颜色（WCAG AA 最小值，纯色信号，问题组合，纯白/纯黑标记）
- 语义化 HTML 和结构（标题，button 与 div，标签，alt 文本，ARIA 纪律）
- 键盘导航和焦点（可达性，Tab 顺序，可见的焦点，跳过链接）
- 动效、表单和杂项（`prefers-reduced-motion`，闪烁限制，错误明确性，点击目标大小）

将发现的问题作为分类列表报告。

### 通道 2：AI 套路检查 (Pass 2: AI slop check)

运行完整的 `ai-slop-check` 审查：
- 过度的渐变
- 表情符号作为装饰
- 带有左边框强调的圆角（作为默认值使用）
- 手绘 SVG 插画
- 将被滥用的字体作为默认值（Inter, Roboto, Arial, Fraunces，无修饰的系统字体栈）
- 纯白和纯黑
- 随机编造的颜色
- 随机的间距值

报告发现的问题。

### 通道 3：层级与韵律审查 (Pass 3: Hierarchy and rhythm review)

运行完整的 `hierarchy-rhythm-review`：
- **层级 (Hierarchy)：** 第一/第二/第三级区分、尺寸、颜色、字重、位置、密度、5 秒测试
- **韵律 (Rhythm)：** 间距比例尺的纪律、字号阶梯的纪律、重复、战略性变化、调色板纪律、区域结构、对齐

报告发现的问题。

### 通道 4：交互状态审查 (Pass 4: Interaction states pass)

运行完整的 `interaction-states-pass`：
- 清点交互元素
- 对于每一个元素：默认、悬停 (hover)、激活 (active)、禁用 (disabled)、焦点 (focus)、加载中 (loading)
- 过渡动效（状态变化为 0.15–0.3 秒，进入/退出更长，尊重 `prefers-reduced-motion`）
- 操作反馈（成功/错误确认，状态可见性）

报告发现的问题。

## 阶段 3：汇总，去重，排定优先级 (Phase 3: Aggregate, deduplicate, prioritize)

等待所有四个代理完成。将发现汇总到一个列表中。

**去重。** 如果两个代理标记了同一个问题（例如，“焦点环被移除”在无障碍和交互状态中都出现了），则合并为一个条目。

**优先级排序。** 将发现分为：

1. **阻碍因素 (Blockers)** —— 无障碍性故障（对比度低于 WCAG，缺少键盘支持，焦点环被移除，缺少标签）。这些问题会毁了真实用户的体验；修复它们所有。
2. **质量问题 (Quality issues)** —— AI 套路、层级崩坏、缺少交互状态。这些问题使设计变得廉价；修复它们所有。
3. **精修建议 (Polish recommendations)** —— 更微妙的改进（建议颜色调子改变，间距比例收紧）。在范围内时应用；在范围外时标记。

## 阶段 4：修复 (Phase 4: Fix)

直接修复每一个阻碍因素和每一个质量问题。如果精修建议不与用户表明的方向冲突，应用它们。

对于模棱两可的修复（例如，“设计使用了 Inter，但用户没有给出品牌字体的偏好”），挑选一个站得住脚的默认项，并在总结中记录此选择，以便用户可以覆盖。

对于那些明显是误报或超出范围的发现（例如，“第三方嵌入组件有低对比度 —— 但那是一个第三方组件”），记录下来并跳过。

## 阶段 5：再次验证 (Phase 5: Re-verify)

修复后，对高风险区域进行快速重新检查：
- 对比度的修复是否保持了视觉风格，还是冲淡了品牌色彩？
- 增加的焦点环是否与相邻内容重叠？
- 调整的层级是否让主要的 CTA 感觉确实是主要操作？

如果看起来不对劲，修复它。如果你不确定，标记出来让用户审查。

## 阶段 6：最终总结 (Phase 6: Final summary)

简明扼要地报告：

- **最终结论 (Verdict)** —— “准备发布” / “在用户审查标记的决定后准备发布” / “在精修有意义之前需要更多迭代”
- **修复的阻碍因素** —— 按分类计数（无障碍 / AI 套路 / 层级 / 交互）
- **应用的精修** —— 按分类计数
- **开放性决策** —— 需要用户签字同意的判断（字体选择、颜色调子变化、层级强调程度）
- **超出范围** —— 任何你注意到但没碰的东西（例如，文案修改，内容添加，新功能）

保持总结简短。如果用户想要详细信息，他们可以询问。**简短的总结 —— 只包含注意事项和后续步骤。**
