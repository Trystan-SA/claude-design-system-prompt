# 制作原型：交互式可点击原型

构建一个可工作的交互原型——可点击、可导航、具有真实状态和反馈。当用户要求制作原型、设计稿、演示或"让它交互起来"时使用此技能。

**原型是交互式的。** 用 `<a>` 标签串联起来的静态截图不算。原型的意义在于通过真实的交互来测试流程——点击、输入、验证、成功、失败。

## 阶段 1：发现

构建前确认：

- **流程。** 有哪些屏幕？入口点是什么？目标状态是什么？以列表形式映射。
- **保真度。** 高保真（真实视觉、真实组件、真实感觉）还是中保真（线框图级别，专注于流程而非打磨）？
- **设备框架。** 桌面浏览器？iOS 框架？Android 框架？macOS 窗口？使用相应的起始组件。
- **变体。** 用户想要一个流程还是多个流程进行对比？
- **品牌 / 设计系统。** 始终确认。如果没有，先调用 `frontend-aesthetic-direction`。
- **示例数据。** 用什么看起来真实的内容来填充屏幕？避免使用 Lorem ipsum。

## 阶段 2：映射屏幕和状态

构建前，写下流程：

```
屏幕：
  1. 欢迎 — "开始使用" 行动号召（CTA）→ 跳转到 2
  2. 邮箱输入 — 验证格式 → 有效则跳转到 3，无效则显示错误
  3. 验证 — "检查你的邮箱" + "跳过" → 跳转到 4
  4. 个人资料 — 姓名、头像上传 → 跳转到 5
  5. 成功 — "你已加入" + "开始使用" → 跳转到 1（循环演示）

状态：
  - currentScreen: 1
  - email: ""
  - emailError: null
  - name: ""
```

将此作为注释块放在文件顶部，以便用户看到你的计划。

## 阶段 3：逐屏构建

对于每个屏幕：

1. **挂载到 DOM 中** — 使用显示切换（`display: none` / `display: block`）或 React 状态在单页内切换屏幕。
2. **高保真视觉。** 匹配设计系统。使用真实组件，而非通用方框。
3. **真实内容。** 看起来可信的示例数据——真实姓名、真实产品文案、真实数字。
4. **每屏一个主要行动号召（CTA）。** 次要操作更小并弱化处理。

使用正确的设备框架：

- **iOS** — `copy_starter_component`，`kind: "ios_frame.jsx"`
- **Android** — `copy_starter_component`，`kind: "android_frame.jsx"`
- **macOS 窗口** — `copy_starter_component`，`kind: "macos_window.jsx"`
- **浏览器** — `copy_starter_component`，`kind: "browser_window.jsx"`

框架保持固定；原型在其内部运行。

## 阶段 4：连接交互

真正的原型应该**每个交互都连接好**，而不仅仅是理想路径：

- **导航。** 点击主要行动号召（CTA）跳转到下一屏。返回按钮跳转到上一屏。状态在屏幕间持续保持。
- **表单验证。** 空提交 → 内联错误。格式错误 → 与字段关联的具体错误。输入有效 → 继续。
- **加载中状态。** 异步操作（注册、保存）显示加载指示器。按钮在请求期间禁用以防止重复提交。
- **成功反馈。** Toast 通知、内联确认或确认操作的页面过渡。
- **错误反馈。** 错误信息清晰且与导致错误的字段或操作相关联。
- **状态变化。** 切换、选择、筛选——所有操作立即更新 UI。

如果原型是一个小切片，用 `setTimeout` 模拟异步工作以模拟延迟。不要因为工作是模拟的就跳过加载中状态——加载中状态本身就是原型要测试的内容之一。

## 阶段 5：连接子状态

许多真实流程具有有意义的子状态：

- **选择状态** — 列表中哪个项目被选中
- **筛选 / 排序状态** — 数据当前的排列方式
- **弹窗 / 下拉状态** — 打开或关闭
- **表单状态** — 值、错误、脏值/原始值

使这些都是响应式的。如果用户点击筛选标签，列表重新渲染。如果他们打开弹窗，焦点移到弹窗且 Escape 关闭它。

## 阶段 6：持久化重要内容

某些状态应在页面重新加载后保留：

- **当前屏幕** — 存储在 `localStorage` 中，加载时恢复
- **表单草稿** — 如果用户在流程中途刷新，应该能从中断处继续
- **调整值** — 参见 `make-tweakable` 技能

在迭代设计过程中刷新是用户最常见的操作之一。无法在重新加载后保留的状态会让原型感觉是坏的。

## 阶段 7：验证

在预览中走完整个流程：

- 每个行动号召（CTA）都有去向
- 每个表单都有验证
- 每个错误都清晰且可恢复
- 每个异步操作都有反馈
- 每个状态变化都可见
- 键盘导航正常工作（Tab 切换、Enter 提交、Escape 关闭弹窗）
- 焦点状态可见

如果你无法验证某个 UI 行为，在总结中明确说明，而不是声称成功。

## 阶段 8：变体（如果被要求）

如果用户要求了流程、布局或视觉处理的变体，将其展示为：

- **调整项** — 在浮动面板中切换（参见 `make-tweakable`）
- **画布** — 通过 `design_canvas.jsx` 起始组件并排展示多个变体
- **原型中的切换** — 用户在选项之间点击切换

不要在项目中散布 v1.html / v2.html / v3.html。一个文件，多个变体。

简要总结：哪些流程可以工作，哪些是模拟的（例如，"提交调用的是 setTimeout 模拟"），哪些留给用户决定。
