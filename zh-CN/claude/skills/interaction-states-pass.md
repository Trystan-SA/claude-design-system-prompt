# 交互状态审查 (Interaction States Pass)

验证每个交互元素是否都具备一套完整的状态（默认 default、悬停 hover、激活 active、禁用 disabled、聚焦 focus），以及合适的过渡动效和反馈。补充缺失的内容。

**没有状态反馈的交互元素会让人感觉坏掉了。** 一个没有 hover 状态的按钮看起来就像个标签。一个看起来像启用了但点击没反应的 disabled 按钮会让人感觉坏掉了。移除 focus 环会将键盘用户拒之门外。

这项技能是设计呈现给用户之前的安全网。

## 阶段 1：清点交互元素 (Phase 1: Identify interactive elements)

遍历设计并清点每一个交互元素。类别包括：

- **按钮 (Buttons)** — `<button>`、任何 `role="button"` 的元素、任何调用点击处理程序的元素
- **链接 (Links)** — `<a>`、任何导航或打开外部内容的元素
- **表单输入 (Form inputs)** — `<input>`、`<textarea>`、`<select>`、文件选择器、范围滑块、颜色选择器
- **开关 (Toggles)** — 复选框、单选按钮、自定义开关
- **作为链接的卡片或行 (Cards or rows that act as links)** — 表格中可点击的行、网格中可点击的卡片
- **导航项 (Navigation items)** — 选项卡、侧边栏链接、面包屑
- **自定义小部件 (Custom widgets)** — 下拉菜单、手风琴菜单、模态框、弹出框

对于每个元素，你需要在阶段 2 中验证完整的状态集。

## 阶段 2：逐个元素进行状态验证 (Phase 2: Per-element state verification)

对于每个交互元素，检查以下六个方面。标记你发现的所有内容，包括边缘情况 —— 记下你的置信度，而不是默默地丢弃你不确定的发现。

### 1. 默认（静止）状态 (Default (resting) state)

该元素在静止时看起来具有明显的交互性：
- 按钮有背景填充、边框，或两者兼有 —— 与正文明显不同
- 链接明显是链接 —— 颜色 + 下划线，或清晰的视觉处理
- 表单输入框具有可见的边框或填充

标记那些看起来像静态文本，仅在 hover 时才显示交互性的元素。有些用户永远无法 hover（触摸设备、键盘用户）。

### 2. 悬停状态 (Hover state)

光标悬停时的视觉变化。至少要有颜色的改变。更好的是：颜色 + 阴影 + 轻微的形变（例如，`translateY(-2px)`）。

标记缺失的 hover 状态。不要降低透明度作为按钮的 hover 状态 —— 这会使它们看起来像被禁用了。

### 3. 激活 / 按下状态 (Active / pressed state)

点击时的视觉变化 —— 通常是更深的颜色、轻微的缩小 (`transform: scale(0.98)`)，或者如果 hover 使元素抬起，则返回基线。active 状态向用户确认在操作完成之前已注册点击。

### 4. 禁用状态 (Disabled state)

明显的禁用特征：较低的透明度（约 0.6）、没有 hover 效果、`cursor: not-allowed`、中性或柔和的颜色。disabled 状态看起来必须既不同于 default 状态也不同于 hover 状态。

如果按钮因为用户没有满足某些条件（例如，“填写所有必填字段”）而被禁用，请提供一个解释*为什么*的指示器 —— 比如工具提示、内联消息或 `title` 属性。一个默默禁用的按钮是一个让人沮丧的陷阱。

### 5. 聚焦状态 (Focus state)

为键盘用户提供可见的 focus 环。使用 `:focus-visible` 而不是 `:focus`，这样 focus 环就会在键盘导航时显示，而不会在每次鼠标点击时都显示。

要求：
- 聚焦环相对于相邻背景必须可见（最低对比度 3:1）
- 聚焦环至少 2px 厚，并带有 2px 的偏移量 (offset)
- **绝对不要**在没有替代方案的情况下使用 `outline: none`

### 6. 加载中状态（适用于触发异步工作的元素） (Loading state)

对于提交表单、保存数据或以其他方式等待网络调用的按钮：
- 点击后立即禁用按钮（防止二次提交）
- 将标签替换为加载动画（spinner）或“加载中...”文本
- 完成后重新启用并恢复标签

对于渲染时获取数据的元素：在等待时显示骨架屏、加载动画或进度指示器。

## 阶段 3：验证过渡动效 (Phase 3: Verify transitions)

每个状态的改变都应该是平滑过渡的，而不是瞬间切换 (snapped) 的：

```css
button {
  transition: background 0.2s ease, transform 0.2s ease, box-shadow 0.2s ease;
}
```

检查过渡持续时间：
- **0.15–0.3s** 用于状态变化（hover、focus、active）—— 感觉响应迅速
- **0.3–0.5s** 用于进入/退出（模态框、抽屉、吐司）—— 感觉沉稳
- **避免** 在微交互中使用慢于 0.5s 的过渡；它们会让人觉得卡顿
- **避免** 0s 的过渡或没有过渡；状态改变会让人觉得突兀或坏掉了

将包含大量动效的过渡包装在 `@media (prefers-reduced-motion: reduce)` 中，以缩短或移除它们，照顾那些偏好减少动效的用户。

## 阶段 4：验证操作反馈 (Phase 4: Verify feedback for actions)

对于用户采取的每一个操作，结果都应该是可见的：

- **表单提交成功** — 吐司、内联消息，或带确认信息的页面重定向
- **表单提交失败** — 清晰的错误消息，如果是特定字段的错误，需要与该字段关联
- **验证错误** — 当字段失去焦点或提交时显示；当用户修复问题时清除
- **状态改变（开关开启/关闭，项目添加到列表）** — 立即发生的视觉变化，可选带有简短的动画

标记静默的成功（“用户提交了，页面没有明显的变化”）和静默的失败（“用户提交了，什么也没发生，没有显示错误”）—— 这两者都会让人感觉出了故障。

对于状态的可见性：
- 导航中当前页面或选项卡在视觉上应清晰可辨
- 列表中被选中的项目在视觉上应清晰可辨
- 激活的筛选器或排序在视觉上应清晰可辨

## 阶段 5：应用修复 (Phase 5: Apply fixes)

添加每一个缺失的状态或反馈元素。使用设计系统中的令牌来设置颜色和时间。如果设计系统没有定义，请使用合理的默认值：

- 悬停 (Hover)：比默认状态暗 10-15%（如果设计使用现代 CSS，则可以使用 `color-mix`）
- 激活 (Active)：再暗 10%，或 `transform: scale(0.98)`
- 禁用 (Disabled)：opacity 0.6 + `cursor: not-allowed`
- 聚焦 (Focus)：`outline: 2px solid var(--color-primary); outline-offset: 2px`
- 过渡动效 (Transition)：`0.2s ease`

对于无法明确其正确状态的元素（例如，一个切换按钮 —— 什么是它的 active vs. inactive vs. hover-on-active 状态？），做出你的判断并在总结中说明。

## 阶段 6：总结 (Phase 6: Summarize)

汇报：
- 清点出的交互元素
- 添加的缺失状态（按类别计数 —— hover / active / disabled / focus / loading）
- 添加或规范化的过渡动效
- 添加的反馈（吐司、错误消息、加载指示器）
- 任何需要用户审查的判断
