# 无障碍性审计：WCAG 与包容性设计审查 (Accessibility Audit: WCAG and Inclusive Design Review)

审查当前设计在对比度、语义结构、键盘导航、动效和表单方面的无障碍性问题。修复发现的任何问题。**良好的无障碍性就是好的设计 —— 它让每个人都受益。**

## 阶段 1：确定要审查的范围 (Phase 1: Identify the surface to audit)

确定要审查的内容。按优先级排序：

1. 用户刚刚编辑或询问的 HTML 文件。
2. 项目中最近修改的设计文件。
3. 如果不清楚，询问用户要审计哪个文件。

从头到尾阅读文件。注意：正在使用的框架或组件库、预期的部署无障碍性级别（WCAG AA 是标准默认值），以及用户提出的任何限制条件。

## 阶段 2：并行启动四个审查代理 (Phase 2: Launch four review agents in parallel)

使用 ${AGENT_TOOL_NAME} 工具在一条消息中同时启动所有四个代理。将完整的文件内容传递给每个代理，以便其拥有完整的上下文。

明确指示每个代理：报告发现的每一个问题，包括边缘情况和低严重性的问题，并附上置信度和严重程度的估计。覆盖范围是代理的工作；过滤则发生在汇总阶段（阶段 3）。

### 代理 1：对比度与色彩 (Agent 1: Contrast and Color)

对于每个文本和 UI 元素：

1. **验证文本对比度。** 普通文本（18px 以下）需要 4.5:1；大文本（18px+ 粗体或 24px+）需要 3:1；UI 组件（按钮、图标、聚焦环）需要 3:1。计算你能解析的任何颜色对的实际比例（解析出的十六进制值、追溯到源头的令牌）。标记每一对未通过的颜色，并附上比例和要求的最小值。
2. **检查仅靠颜色传递的信息。** 标记任何仅靠颜色传达的状态 —— 没有图标的绿/红、没有下划线的蓝色链接、没有图例或文本标签的图表。
3. **检查困难的颜色组合。** 红色+绿色（最常见的色盲）、相似亮度下的蓝色+黄色、白色背景上的浅灰色、相似亮度下有色背景上的有色文本。
4. **检查白色和黑色。** 标记在 `#000000` 上的纯 `#FFFFFF`。首选微妙调整过色调的颜色（例如，`#FAFAFA` / `#1A1A1A`）—— 尽管这是风格要求，而非 WCAG，所以作为建议标记。

### 代理 2：语义化 HTML 与结构 (Agent 2: Semantic HTML and Structure)

对于文档结构：

1. **标题层级。** 只有且仅有一个 `<h1>`。不能跳级（不要从 `<h2>` 直接到 `<h4>`）。标题用于描述内容，而不是改变视觉大小。
2. **角色匹配正确的元素。** 使用 `<button>` 而不是 `<div onclick>`。使用 `<a href>` 而不是样式像链接的 `<div>`。使用 `<label for="id">` 链接到 `<input id="id">`。使用 `<nav>`、`<main>`、`<article>`、`<section>`、`<aside>` 定义地标区域（landmarks）。
3. **每个有意义的图像都有 Alt 文本。** 装饰性图像使用 `alt=""` 以便屏幕阅读器跳过它们。有意义的图像描述它们传达的内容，而不是它们是什么（是 `alt="Wireless headphones, side view"` 而不是 `alt="product"`）。
4. **表单输入标签。** 每个输入都有一个关联的 `<label>`（如果视觉上没有标签，则使用 `aria-label`）。只有占位符文本不是标签 —— 当用户输入时，占位符就会消失。
5. **在语义化 HTML 有效时避免使用 ARIA。** 如果一个 `<div>` 本可以是 `<button>`，标记其上的 `role="button"`。ARIA 是一个补丁，而不是默认选项。

### 代理 3：键盘导航与焦点 (Agent 3: Keyboard Navigation and Focus)

对于每个交互元素：

1. **键盘可访问。** 所有可点击的内容都必须可以通过 Tab 键触达。只有悬停（hover）才能出现的菜单、无法用键盘打开的模态框、需要鼠标悬停的下拉菜单都会判定为失败。
2. **逻辑 Tab 顺序。** Tab 序列应该遵循阅读顺序（从上到下，从左到右）。标记显式声明 `tabindex` 大于 0 的值（它们扭曲了自然顺序）。
3. **键盘交互模式。** 模态框可以通过 Escape 键关闭。下拉菜单通过 Enter/Space 打开，并通过方向键导航。表单可以通过字段上的 Enter 键提交。
4. **可见的聚焦环。** 标记任何没有替代方案的 `outline: none`。替代方案应该是可见的，并且与相邻背景满足 3:1 的对比度。首选 `:focus-visible` 而不是 `:focus`。
5. **跳过链接 (Skip links)。** 对于有大量重复导航的页面，建议添加一个“跳过导航到主要内容 (Skip to main content)”的链接，作为第一个可聚焦的元素。

### 代理 4：动效、表单与其他 (Agent 4: Motion, Forms, and Misc)

1. **尊重 `prefers-reduced-motion`。** 超过几百毫秒的动画和过渡应该有一个 `@media (prefers-reduced-motion: reduce)` 块来缩短或移除它们。
2. **无闪烁内容。** 任何每秒闪烁超过 3 次的内容都会引发光敏性癫痫。自动播放的视频、频闪效果、快速循环 —— 标记并要求提供暂停控制。
3. **表单错误。** 每个错误消息都必须是具体的（“邮箱地址无效”而不是“无效”），与它的字段相关联（视觉上相邻，并通过 `aria-describedby`），并由屏幕阅读器播报。
4. **必填字段。** 使用文本和/或图标加上 `required` 属性来标记，不能仅靠颜色。
5. **输入类型和自动完成。** 使用 `<input type="email">` 输入邮箱，使用 `type="tel"` 输入电话，使用 `autocomplete` 属性进行自动填充。这提高了移动端键盘体验和无障碍性。
6. **触摸目标尺寸。** 在触摸界面上，按钮、链接和可点击区域应至少为 44px × 44px。

## 阶段 3：汇总与修复 (Phase 3: Aggregate and fix)

等待所有四个代理完成任务。将他们的发现汇总到一个列表中，如果多个代理标记了同一问题，请去重。

直接修复每个问题。对于模棱两可的情况（例如，“这个对比度是 4.4:1，非常接近通过了”），不管怎样也要修复 —— 无障碍性是底线，而不是上限。

如果某个发现明显是误报，或是超出范围（例如，代理标记了你无法修改的第三方嵌入内容），记录并跳过。不要对发现进行争论 —— 继续做下去。

完成时，总结：
- 发现的问题（按类别分类：对比度 / 语义 / 键盘 / 动效-表单）
- 已修复的问题
- 留给用户解决的问题（第三方内容、模棱两可的情况、超出无障碍性范围的设计决定）
